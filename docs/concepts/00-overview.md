# 공학적 개념 가이드 - 개요

> *"모든 복잡한 시스템의 구축은 기초에 대한 깊은 이해에서 시작됩니다."*

폐쇄망(Air-Gapped) 환경에서 Kubernetes 클러스터를 성공적으로 배포하고 운영하려면 단순히 명령어를 따라하는 것 이상의 지식이 필요합니다. 왜 특정 설정이 필요한지, 문제가 발생했을 때 어떻게 원인을 추적해야 하는지, 시스템이 어떻게 동작하는지에 대한 **근본적인 이해**가 필요합니다.

이 문서 시리즈는 그러한 기초 지식을 체계적으로 제공합니다. Linux 커널의 기본 개념부터 분산 시스템 이론, 최신 eBPF 기술까지 - Kubespray를 사용한 오프라인 Kubernetes 배포에 필요한 모든 공학적 개념을 다룹니다.

---

## 이 장에서 다루는 내용

이 개요 문서를 읽고 나면 다음을 이해할 수 있습니다:

- **학습 로드맵**: 전체 문서 시리즈의 구조와 학습 순서
- **대상 독자**: 각 역할별 권장 학습 경로
- **사전 요구 지식**: 이 문서들을 학습하기 전 필요한 기본 지식
- **문서 활용 가이드**: 효과적인 학습을 위한 권장 방법
- **프로젝트 개요**: Kubespray 오프라인 설치 프로젝트의 전체 아키텍처
- **핵심 용어집**: 자주 사용되는 전문 용어와 약어 정리

## 1. 왜 기초 개념이 중요한가?

### 1.1. 단순 매뉴얼을 넘어서

많은 엔지니어들이 "명령어만 알면 된다"고 생각합니다. 하지만 실제 운영 환경에서는 예상치 못한 문제가 항상 발생합니다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                       왜 기초 개념 학습이 필요한가?                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   시나리오 1: "Pod가 생성되지 않아요"                                        │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  기초 없이: "왜 안 되지?" → 구글링 → 복사-붙여넣기 → 다른 문제 발생    │   │
│  │                                                                       │   │
│  │  기초 있으면:                                                          │   │
│  │  1. kubectl describe pod → 이벤트 확인 (스케줄링? 이미지? 리소스?)     │   │
│  │  2. kubelet 로그 → CRI 호출 확인                                      │   │
│  │  3. containerd 로그 → 이미지 풀 실패 원인 파악                         │   │
│  │  4. 레지스트리 프록시 → 실제 원인: 이미지 캐싱 안 됨                    │   │
│  │  → 근본 원인 해결                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   시나리오 2: "네트워크가 간헐적으로 끊겨요"                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  기초 없이: 재부팅 → 임시 해결 → 반복                                  │   │
│  │                                                                       │   │
│  │  기초 있으면:                                                          │   │
│  │  1. Cilium 상태 확인 → eBPF 맵 확인                                   │   │
│  │  2. BPF 프로그램 검사 → 정책 충돌 발견                                 │   │
│  │  3. NetworkPolicy 검토 → 잘못된 규칙 수정                              │   │
│  │  → 영구적 해결                                                         │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   시나리오 3: "etcd가 불안정해요"                                           │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  기초 없이: 노드 교체 시도 → 쿼럼 손실 → 클러스터 다운                  │   │
│  │                                                                       │   │
│  │  기초 있으면:                                                          │   │
│  │  1. Raft 합의 알고리즘 이해 → 쿼럼 계산                                │   │
│  │  2. etcdctl member list → 현재 상태 확인                              │   │
│  │  3. 안전한 멤버 교체 절차 수행                                         │   │
│  │  → 무중단 복구                                                         │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

> **📘 개념 (Concept)**: 기초 개념을 이해하면 **근본 원인 분석(Root Cause Analysis)** 능력이 생깁니다. 이는 단순히 증상을 해결하는 것이 아니라, 문제의 진짜 원인을 찾아 영구적으로 해결하는 능력입니다.

### 1.2. 기술의 상호 연결성

Kubernetes는 독립된 기술이 아닙니다. 수십 가지 기술이 유기적으로 연결되어 작동합니다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      Kubernetes 기술 스택의 계층 구조                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│                    ┌───────────────────────────────────────┐                │
│   애플리케이션 계층 │  Deployments, Services, Ingress       │                │
│                    │  ConfigMaps, Secrets, RBAC            │                │
│                    └───────────────────────┬───────────────┘                │
│                                            │ Kubernetes API                 │
│                    ┌───────────────────────┴───────────────┐                │
│   오케스트레이션   │  kube-apiserver, kube-scheduler       │                │
│   계층            │  kube-controller-manager, etcd         │                │
│                    └───────────────────────┬───────────────┘                │
│                                            │ CRI, CNI                       │
│                    ┌───────────────────────┴───────────────┐                │
│   컨테이너 계층    │  containerd, runc                     │                │
│                    │  Cilium (eBPF CNI)                    │                │
│                    └───────────────────────┬───────────────┘                │
│                                            │ syscall                        │
│                    ┌───────────────────────┴───────────────┐                │
│   운영체제 계층    │  Linux Kernel                         │                │
│                    │  namespaces, cgroups, netfilter       │                │
│                    │  eBPF subsystem                       │                │
│                    └───────────────────────┬───────────────┘                │
│                                            │ hardware I/O                   │
│                    ┌───────────────────────┴───────────────┐                │
│   하드웨어 계층    │  CPU, Memory, Network, Storage        │                │
│                    └───────────────────────────────────────┘                │
│                                                                             │
│   → 각 계층을 이해해야 문제 발생 시 어느 계층에서 원인을 찾아야 하는지 알 수 있음  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

> **💡 팁 (Tip)**: 문제 해결 시 **"위에서 아래로"** 또는 **"아래에서 위로"** 접근법을 사용하세요. 애플리케이션 레벨에서 시작해 커널까지 내려가거나, 네트워크 연결부터 확인해 올라가는 방식입니다.

### 3.2. 경험 수준별 권장 학습

#### 입문자 (Kubernetes 경험 없음)

```
순서: 01 → 02 → 03 → 04 → 05 → 06 → 07 (전체 순차 학습)
```

- 모든 기초 문서(01-03)를 먼저 완료하세요
- 이해가 안 되는 부분이 있으면 건너뛰지 말고 추가 자료를 찾아보세요
- 각 문서의 실습 예제를 직접 따라해 보세요

#### 중급자 (Kubernetes 운영 경험 있음)

```
순서: 00(이 문서) → 05 → 06 → 07
```

- 기초 문서(01-04)는 참조용으로 필요할 때 확인
- 분산 시스템(05)과 eBPF(06) 문서에 집중
- 오프라인 인프라(07) 문서로 실무 적용

#### 특정 주제만 필요한 경우

| 주제 | 권장 문서 |
|------|----------|
| 네트워킹 문제 해결 | 03 → 06 |
| etcd/고가용성 | 05 |
| 컨테이너 이미지 문제 | 02 → 07 |
| 성능 최적화 | 01 → 06 |
| 오프라인 환경 구축 | 07 (+ 필요시 02, 03) |

> **⚠️ 주의 (Warning)**: 심화 문서(04-06)를 제대로 이해하려면 기초 문서(01-03)의 개념이 필요합니다. 시간이 촉박하더라도 최소한 기초 문서를 빠르게 훑어보는 것을 권장합니다.

## 6. 프로젝트 개요: Kubespray 오프라인 설치

### 6.1. 왜 오프라인 설치가 필요한가?

많은 기업 환경, 특히 금융, 의료, 공공, 국방 분야에서는 보안상의 이유로 **폐쇄망(Air-Gapped Network)** 을 운영합니다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        폐쇄망이 필요한 이유                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   1. 보안 규정 준수                                                         │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 금융: ISMS-P, PCI-DSS                                             │  │
│   │  • 의료: HIPAA, 개인정보보호법                                        │  │
│   │  • 공공: 국가정보원 보안 지침                                         │  │
│   │  • 국방: 국방부 사이버보안 정책                                       │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   2. 외부 의존성 최소화                                                     │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 외부 레지스트리 장애 시에도 서비스 지속                            │  │
│   │  • Docker Hub rate limit 회피                                        │  │
│   │  • 공급망 공격(Supply Chain Attack) 방지                             │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   3. 재현 가능한 배포                                                       │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │  • 버전이 고정된 이미지와 바이너리 사용                               │  │
│   │  • 동일한 환경을 여러 번 배포 가능                                    │  │
│   │  • 감사(Audit) 추적 용이                                              │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
